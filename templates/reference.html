Responsible Use

PhantomFuzzer sends real, active requests to the target — this is fundamentally different from passive OSINT tools. Only use it against applications you own or have explicit written authorization to test (e.g. your own dev/staging environment, or a scope-defined bug bounty program). It is a detection tool — it sends known probe payloads and analyzes responses for anomalies. It never attempts to extract data, spawn a shell, or exploit a confirmed finding.

SQL Injection Detection
TechniqueHow it works
Error-basedSends characters that break SQL syntax (', --), checks response for known DB error signatures
Boolean-based blindSends a TRUE condition (' OR '1'='1) and FALSE condition (' AND '1'='2), flags if responses differ significantly — a safe backend treats both as inert text
Time-based blindSends SLEEP(5)/WAITFOR DELAY/pg_sleep(5), flags if the response takes measurably longer than baseline
XSS Detection

Sends payloads containing HTML-significant characters (<, >, ", ') and checks whether they're reflected verbatim (vulnerable) or HTML/URL-encoded (safe). Payloads without those characters are excluded from this check — escaping can't change what it has nothing to escape.

Command Injection Detection

Sends shell-chaining payloads (; echo MARKER, | echo MARKER, $(echo MARKER)) with a unique marker string, and time-based payloads (; sleep 5). Flags a match only if the marker appears WITHOUT the full injected payload also appearing (raw or encoded) — otherwise, an app that simply echoes input back would falsely appear vulnerable.

Path Traversal

Sends ../../../../etc/passwd-style payloads and checks for known file-content signatures (e.g. the header line of /etc/passwd or a Windows win.ini section marker).

Server-Side Template Injection (SSTI)

Sends arithmetic template expressions ({{7*7}}, ${7*7}, #{7*7}) and checks if they were EVALUATED (produces 49) rather than reflected literally — the classic technique for detecting when a template engine executes user input.

Confidence Levels

High — strong, specific signal (known error string, marker executed cleanly, template expression evaluated).
Medium — behavioral difference consistent with a vulnerability but with more room for coincidence (boolean-based response differences, borderline timing).
Low — worth a manual look, but a weak/generic signal (unexpected server error on fuzz input).

No automated tool can prove a vulnerability with certainty from black-box testing alone — always manually verify findings before reporting or remediating.
